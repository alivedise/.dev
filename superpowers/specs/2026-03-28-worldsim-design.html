WorldSim -- Persistent Virtual World Simulation ​

A backend-focused learning project built in Go. Autonomous agents inhabit a tile-based world, gather resources, interact, form settlements, and develop an emergent economy. The backend is the product; observation happens through a minimal terminal dashboard.

Goals ​

  • Cover the full backend skill spectrum: system design, Go mastery, infrastructure (Docker, Postgres, Redis, gRPC, message queues), data modeling, and observability
  • Weekend-sized milestones that each produce something observable
  • Modular monolith architecture with clean package boundaries, extracting services when motivated by real needs

Architecture ​

Approach: Modular Monolith, Extract Later ​

Single Go binary with cleanly separated internal packages. In-process communication initially. Services are extracted into separate processes when scaling or deployment needs motivate it (milestone 12+).

Project Structure ​

worldsim/
  cmd/
    server/main.go            # simulation server entry point
    dashboard/main.go          # terminal dashboard client
  internal/
    world/                     # map, tiles, terrain, resource regen
    agent/                     # agent state, lifecycle, reproduction
    ai/                        # GOAP planner, perception, memory, decisions
    economy/                   # trade, markets, order book, currency
    craft/                     # recipe engine, crafting resolution
    config/                    # YAML loaders for resources, recipes, terrains
    simulation/                # tick loop, action resolution, conflict handling
    storage/                   # Postgres persistence, snapshots, migrations
    cache/                     # Redis hot state, pub/sub for events
    event/                     # event types, event bus (in-process first)
    api/                       # gRPC service definitions (added when extracting)
  config/
    resources.yaml
    recipes.yaml
    terrains.yaml
    world.yaml                 # map size, tick rate, spawn rules, etc.
  migrations/                  # Postgres schema migrations
  docker-compose.yaml          # Postgres, Redis, Grafana, Prometheus
  Makefile

Concurrency Model ​

  • Simulation tick loop runs on a single goroutine for world state consistency -- no concurrent writes during a tick.
  • Agent perception and decision-making parallelized across a worker pool within each tick (agents don't write to world state during the think phase).
  • Action resolution is sequential within a tick for deterministic conflict handling.
  • Snapshot persistence runs on a separate goroutine, operating on a copy of world state.
  • Dashboard connects via Redis pub/sub, fully decoupled from the simulation loop.

Persistence ​

  • Postgres: world snapshots (full serialized state every N ticks), agent history (births, deaths, lineage), economy logs (trades, currency events), settlement records. Schema versioned via golang-migrate.
  • Redis: current world state for dashboard reads (tiles, agent positions, inventories). Updated each tick via pipeline writes. Pub/sub channels for real-time events (agent_action, trade, death, settlement_founded).

Observability ​

  • Prometheus metrics: tick duration, agent count, resource totals, trades per tick, settlement count, memory usage.
  • Grafana dashboards for simulation health and emergent pattern visualization.
  • Structured logging (zerolog or slog) with tick number and agent ID context.

Terminal Dashboard (bubbletea) ​

  • Reads from Redis, not from the simulation process.
  • ASCII grid view of the world (terrain + agent density).
  • Sidebar: population stats, economy summary, top settlements.
  • Agent inspector: select an agent, see its goals, inventory, memories, relationships.
  • Event log: scrollable feed of recent events.

Docker Compose (from milestone 1) ​

  • worldsim-server: the simulation binary
  • postgres: persistence
  • redis: hot state + pub/sub
  • prometheus + grafana: observability

World Design ​

Map ​

2D grid of tiles, fixed size to start (e.g., 128x128). Procedural generation using Perlin noise for terrain. Resource distribution follows terrain type.

Tile Properties ​

  • Terrain type (config-driven)
  • Resources (type to quantity, with regeneration rate)
  • Structures (config-driven types like shelter, storage, workshop, market -- built by agents)
  • Occupants (list of agent IDs)

Time Model ​

Discrete ticks, not real-time. Configurable tick rate (e.g., 10 ticks/sec). Each tick advances the full simulation loop. Deterministic and debuggable.

Persistence ​

World state snapshots to Postgres periodically (every N ticks). On restart, load last snapshot and resume.

Resource Regeneration ​

Tiles regenerate resources per tick based on terrain config. Over-harvesting depletes a tile temporarily, creating natural pressure for migration and trade.

Agent Design ​

Attributes ​

  • Position (x, y), health, energy, hunger, age
  • Inventory: resource type to quantity, carry capacity limit
  • Skills: gathering, building, crafting, trading, combat, navigation -- each a float 0-1, improves with use, decays with disuse (logarithmic progression, slow linear decay)
  • Personality traits: curiosity, aggression, sociability, industriousness, risk tolerance -- randomly assigned on a spectrum at spawn
  • Memory: bounded list (~100 entries) of remembered locations, agent encounters, danger zones
  • Relationships: map of agent ID to trust score, updated through interactions
  • Goals: priority queue of active goals (survive, explore, accumulate, build, socialize)

Perception ​

  • Vision radius scales with navigation skill (3-8 tiles)
  • Personal fog-of-war map that decays over time (resources may have changed since last visit)
  • Can detect other agents and read apparent intent (moving toward, carrying resources, building) but not internal state

Decision-Making: Goal-Oriented Action Planning (GOAP) ​

  1. World state assessment: evaluate current needs (hunger, safety, inventory, social)
  2. Goal selection: pick highest-priority unmet goal. Personality shifts priorities.
  3. Action planning: search available actions for a short sequence (max 5 steps) that moves toward the goal. Each action has preconditions and effects.
  4. Execution: execute first action in plan. Re-plan every few ticks or on significant situation change.

Example plan for "build shelter": find wood tile -> move -> gather wood -> find stone tile -> move -> gather stone -> find build site -> build shelter

Skills and Progression ​

  • Using a skill increases it (diminishing returns, logarithmic curve)
  • Unused skills decay slowly
  • Skill level affects action speed and success rate
  • High skill unlocks advanced actions (high crafting = workshops, high trading = better barter rates)

Lifecycle and Reproduction ​

  • Agents age over ticks. Young agents: high energy, low skills. Old agents: high skills, declining health.
  • Reproduction: two agents with high mutual trust, high sociability, on same tile, with surplus food. Offspring inherits blended parent traits with mutation.
  • Death from starvation, old age, or combat. Inventory drops on tile.
  • Population is self-regulating through resource pressure.

Social System ​

  • Trust: starts at 0 for strangers. Increases through successful trades, proximity, cooperation. Decreases from theft, combat, failed trades.
  • Communication: agents on adjacent tiles can signal -- share a memory or a request. Deceptive agents (low sociability, high aggression) can share false information.
  • Groups: agents with high mutual trust who cluster together implicitly form a group. Groups share settlements, defend territory, pool resources. Emergence only -- no explicit "create group" action.
  • Conflict: agents can steal from adjacent agents (success based on combat skill vs target's). Combat: compare combat skills + health, loser takes damage and may drop items.

Memory System ​

  • Bounded to ~100 entries, oldest entries evicted first
  • Each memory: type (resource_location, agent_encounter, danger, settlement), position, tick timestamp, reliability score
  • Shared memories via communication have lower reliability scores
  • Decision-making weights recent, high-reliability memories more heavily

Economy Design ​

Data-Driven Resources and Crafting ​

All resources, crafted goods, and recipes defined in YAML configuration files. The engine operates on abstract types and relationships.

Resource config example:

yaml
resources:
  - id: food
    category: raw
    weight: 1
    terrain_sources: [grass, forest, water]
    regen_rate: 0.3
  - id: wood
    category: raw
    weight: 3
    terrain_sources: [forest]
    regen_rate: 0.1

Recipe config example:

yaml
recipes:
  - id: tools
    category: crafted
    inputs:
      - resource: wood
        quantity: 2
      - resource: stone
        quantity: 1
    ticks: 5
    min_skill: 0.2
    effects:
      gathering_speed: 1.5

Terrain config example:

yaml
terrains:
  - id: forest
    movement_cost: 1.5
    vision_penalty: -2
    resources: [wood, food]

Structure types are also config-driven:

yaml
structures:
  - id: workshop
    build_cost:
      - resource: wood
        quantity: 5
      - resource: stone
        quantity: 3
    build_ticks: 10
    min_skill: 0.3
    effects:
      crafting_speed: 2.0
  - id: market
    build_cost:
      - resource: wood
        quantity: 8
      - resource: stone
        quantity: 5
    build_ticks: 15
    min_skill: 0.4
    effects:
      enables_orders: true

Adding new resources, recipes, terrains, or structures requires only editing YAML files. Different config sets can produce different worlds (fantasy, sci-fi, etc.). Agent AI adapts automatically by evaluating resources based on personal scarcity and utility from the effects field.

Trade System ​

  • Barter: agents assign subjective value based on personal scarcity. Two agents trade when both perceive net gain.
  • Emergent currency: settlements at population threshold can mint token currency backed by stockpile. Other agents accept based on trust in the settlement. Low-stockpile settlements see currency devalue.
  • Markets: market structures on tiles allow agents to post buy/sell orders (resource, quantity, price). Order book data structure. Other agents fill orders.

Settlement Economy ​

  • Shared stockpile with configurable tax rate
  • Stockpile funds shared structure construction (walls, workshops, markets)
  • Successful settlements attract agents (positive feedback loop)
  • Inter-settlement trade via traveling agents (emergent trade routes)

Supply and Demand ​

  • Resource scarcity drives migration
  • Settlement growth depletes surrounding tiles, creating expansion pressure
  • Optional later: seasonal cycles (regeneration rate changes over long tick periods)

Milestones ​

Each milestone is achievable in a weekend (1-2 days) and produces something observable.

M1 -- World Exists ​

Procedural world generation from config. Tiles with terrain and resources. Postgres schema + snapshot persistence. Docker Compose with Postgres and Redis. Load, save, restart, reload.

Skills: Go project structure, Postgres, Docker, schema migrations.

M2 -- Agents Walk ​

Spawn agents on the map. Random movement. Agents in Redis. Bubbletea dashboard shows ASCII grid with updating agent positions. Worker pool for parallel perception.

Skills: Redis, pub/sub, goroutine worker pools, bubbletea TUI.

M3 -- Agents Think ​

GOAP planner with basic goals: eat, explore. Agents perceive surroundings, find food, move toward it, gather. Hunger/energy mechanics. Starvation death.

Skills: GOAP/AI patterns, game loop design, concurrent agent processing.

M4 -- Agents Remember ​

Memory system. Agents remember resource locations, revisit good tiles. Fog-of-war personal map. Observable: patrol routes, clustering around rich areas.

Skills: Data structures (bounded caches), spatial queries.

M5 -- Crafting and Skills ​

Config-driven recipe engine. Agents craft tools. Skill progression. Observable: specialization emerges, tool-makers outperform.

Skills: Config-driven engine design, YAML parsing.

M6 -- Agents Trade ​

Barter between adjacent agents. Subjective value. Trust scores. Observable: agents specialize and trade complementary resources.

Skills: Transaction-like operations, consistency in concurrent systems.

M7 -- Settlements Form ​

Agents with high mutual trust cluster and build. Shared stockpiles, tax. Workshops. Observable: settlements appear, grow, attract.

Skills: Aggregate state, shared resource management.

M8 -- Markets and Currency ​

Order book on market structures. Settlement currency backed by stockpiles. Buy/sell orders. Observable: price discovery, inflation/deflation.

Skills: Order book data structure, financial/economic modeling.

M9 -- Conflict ​

Theft, combat, territorial defense. Aggressive agents raid. Settlements build walls. Observable: conflict zones, arms races.

Skills: Conflict resolution algorithms, game theory.

M10 -- Reproduction and Lineage ​

Agents reproduce, offspring inherit traits with mutation. Generational tracking in Postgres. Observable: trait drift, dominant strategies.

Skills: Relational modeling (lineage graphs), genetic algorithms.

M11 -- Observability ​

Prometheus metrics for all subsystems. Grafana dashboards: population curves, resource maps, trade volume, genetic drift.

Skills: Prometheus, Grafana, structured logging, operational thinking.

M12 -- Extract First Service ​

Economy module becomes its own Go service. gRPC interface. NATS or Kafka for trade events. Service discovery, network boundaries.

Skills: gRPC, message queues, service extraction, distributed systems.
